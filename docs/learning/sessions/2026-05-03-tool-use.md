# Practice Session: Tool Use

## Topic

- 日期：2026-05-03
- 参考主题：https://learn.shareai.run/zh/s02/
- 目标：理解工具层如何把模型意图路由成受控动作，并在 Node.js 中实现最小 tool registry。

## Mechanism

s02 的核心不是工具数量，而是稳定边界：

1. `schema` 给模型看，说明有哪些工具、每个工具需要什么参数。
2. `handler` 给 harness 执行，负责真实世界动作。
3. `tool_result` 是结果回流到 agent loop 的统一出口。

只要这三点稳定，加新工具就应该是“新增一个 tool definition”，而不是改 agent loop。

## Python Reference Notes

教学版用 `TOOL_HANDLERS = {tool_name: handler}` 做 dispatch map。模型返回 tool use 后，loop 只按名称查 handler，执行后把结果作为 tool result 写回 messages。

另一个重点是 `safe_path()`：文件工具不能直接信任模型给出的路径，必须把请求路径解析到 workspace 内，防止 `../` 逃逸。

## Node Mapping

- 模块边界：`ToolRegistry` 负责注册和查找，具体工具放在 `src/tools/`，`AgentRunner` 只调用 registry。
- 模型 provider / capability 考虑：tool definition 暴露 OpenAI-compatible function schema；后续 provider adapter 可以再转换成 Anthropic/Gemini 形状。
- 核心数据结构：`ToolDefinition = { name, schema, execute }`。
- 异步/进程/文件系统考虑：文件工具使用 `fs/promises`；bash 工具使用 `child_process.execFile`，限制 cwd、timeout 和 maxBuffer。
- 错误处理策略：handler 抛错时由 `AgentRunner` 转成 error-flavored tool result，继续让模型观察错误。

## Terminal App Notes

Terminal agent 的工具层要避免把 CLI、模型协议和真实执行揉在一起。今天的实现里：

- CLI 只收发文本。
- AgentRunner 只处理 loop 和 tool result 回填。
- ToolRegistry 只做名字到 handler 的分发。
- 具体工具才碰文件系统和子进程。

这让后续加权限确认、hook、审计日志时有明确插入点。

## Execution Walkthrough

一次“读取 README 前 3 行并总结”的真实执行顺序：

1. `src/index.js`
   - `loadEnvFile()` 读取本地 `.env`。
   - `loadLocalModelConfig(process.env)` 生成 `{ provider, baseUrl, model, apiKey }`。
   - `new LocalModelClient(config)` 创建模型 API adapter。
   - `createAgent({ model, workspaceRoot: process.cwd() })` 创建 agent，并把 workspace root 传给工具层。
   - `startTerminal({ agent, config })` 启动终端交互。

2. `src/config/envFile.js`
   - 解析 dotenv 风格的 `KEY=value`。
   - 只在 `process.env` 里没有同名变量时写入，所以 shell export 的变量优先级更高。

3. `src/config/localModelConfig.js`
   - 把 `MODEL_*` 和兼容的 `LOCAL_MODEL_*` 变量整理成统一模型配置。
   - 当前文件名仍叫 `localModelConfig`，但职责已经演进为通用 provider config，后续可以重命名。

4. `src/agent/createAgent.js`
   - 组装 `AgentRunner`。
   - 注入 `model`、`SYSTEM_PROMPT` 和 `createDefaultTools({ workspaceRoot })`。

5. `src/tools/defaultTools.js`
   - 创建 `PathSandbox`。
   - 注册默认工具：`bash`、`read_file`、`write_file`、`edit_file`。

6. `src/tools/pathSandbox.js`
   - 把模型给出的相对路径解析到 workspace root 下。
   - 拒绝 `../` 等路径逃逸，确保文件工具只能触达仓库内部。

7. `src/tools/filesystemTools.js`
   - 每个工具都是 `{ name, schema, execute }`。
   - `schema` 给模型看，`execute` 给 harness 跑。
   - `read_file` 会先过 path sandbox，再用 `fs/promises.readFile` 读取文本。

8. `src/tools/bashTool.js`
   - 使用 `child_process.execFile('bash', ['-lc', command])` 执行命令。
   - 当前限制了 `cwd`、`timeout`、`maxBuffer`。
   - 权限审批尚未实现，留到 s07。

9. `src/tools/ToolRegistry.js`
   - 提供 `register()`、`get()`、`list()`。
   - `AgentRunner` 只按工具名查 registry，不对具体工具写 `if/else`。

10. `src/cli/terminal.js`
    - 读取用户输入。
    - 调用 `agent.send(message)`。
    - 打印最终回复。
    - CLI 不知道具体模型和工具细节。

11. `src/agent/AgentRunner.js`
    - `send()` 把用户输入写入 `history`。
    - `runLoop()` 调用 `model.chat(messages, { tools: this.tools.list() })`。
    - 如果模型返回 `toolCalls`，先把 assistant message 写入历史。
    - 对每个 tool call 执行 `executeToolCall()`。
    - `executeToolCall()` 通过 `ToolRegistry.get(name)` 找到 handler 并执行。
    - 工具结果被转换成 `{ role: 'tool', tool_call_id, name, content }` 并写回 `messages`。
    - loop 继续，模型第二轮基于 tool result 生成最终回答。

12. `src/model/localModelClient.js`
    - 把内部 tool definition 转成 OpenAI-compatible `tools` 参数。
    - 请求 `{baseUrl}/chat/completions`。
    - 把 provider 原始响应归一化为内部 `ModelResponse`。
    - 对 DeepSeek 的 `reasoning_content` 通过 `providerMetadata` 保存并回放。

核心链路可以记成：

```text
terminal input
-> AgentRunner.runLoop()
-> LocalModelClient.chat(messages, tools)
-> provider returns toolCalls
-> ToolRegistry.get(name)
-> tool.execute(input)
-> tool result message
-> LocalModelClient.chat(messages + tool result)
-> final response
-> terminal output
```

## Experiment

- 命令：`npm test`
- 输入：测试覆盖 path sandbox、read/write/edit、bash cwd、默认工具 schema。
- 输出：工具可以读写编辑 workspace 内文件；路径逃逸被拒绝；bash 在 workspace cwd 执行。
- 观察：s02 之后，agent loop 本身没有为具体工具写 `if/else`，工具增长靠 registry。

真实 provider 验证：

- 命令：`printf '请使用工具读取 README.md 的前 3 行，然后用一句话总结。\n/exit\n' | npm start`
- Provider：DeepSeek `deepseek-v4-flash`
- 结果：模型成功调用 `read_file`，runner 将 tool result 写回后，模型第二轮基于 README 内容生成总结。
- 观察：工具 schema 能被 DeepSeek 接收；provider metadata 回放修复后，多轮 tool call 链路可以跑通。

## Provider Compatibility Note

真实用 DeepSeek 跑工具调用时暴露了一个 provider 差异：thinking/tool 场景返回的 assistant message 可能包含 `reasoning_content`，下一轮请求必须把它原样带回，否则 API 会返回 400。

这说明 `ModelResponse.raw` 只适合调试，不能解决所有协议回放问题。对“下一轮必须带回”的供应商字段，我们需要通过 `providerMetadata` 保存，并在内部 assistant message 转换时写回 provider-compatible 字段。
